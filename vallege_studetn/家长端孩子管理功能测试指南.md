# 家长端孩子管理功能测试指南

## 功能概述

家长端孩子管理功能现在实现了以下流程：
1. 家长输入数据库中真实存在的学生姓名
2. 系统验证学生是否存在
3. 发送添加孩子请求给对应学生
4. 学生登录后可以看到请求并接受/拒绝
5. 学生接受后，家长端才能看到该孩子
6. 状态在两边实时变化

## 测试环境

- 后端端口：3000（教师后端，统一使用）
- 家长端前端端口：8082
- 学生端前端端口：8080

### 测试场景一：家长发送添加孩子请求

1. **家长登录**
   - 在家长端使用家长账号登录
   - 进入"孩子管理"页面

2. **输入学生姓名**
   - 在"孩子姓名"输入框中输入数据库中真实存在的学生姓名
   - 可选：填写请求消息
   - 点击"发送添加请求"按钮

3. **预期结果**
   - 如果学生姓名存在：提示"请求已发送，等待学生确认！"
   - 如果学生姓名不存在：提示"未找到该姓名的学生，请检查姓名是否正确"
   - 如果已经发送过请求：提示"已经发送过请求，请等待学生处理"

4. **查看待处理请求**
   - 发送成功后，在"待处理的请求"列表中可以看到该请求
   - 状态显示为"等待学生确认"

### 测试场景二：学生处理家长请求

1. **学生登录**
   - 在学生端使用该学生账号登录
   - 点击左侧菜单"家长请求"

2. **查看请求**
   - 在"待处理的请求"列表中可以看到家长发送的请求
   - 显示家长姓名、请求消息、发送时间

3. **接受请求**
   - 点击"✓ 接受"按钮
   - 确认操作
   - 预期结果：提示"已接受家长请求"，请求从待处理列表移除

4. **拒绝请求**
   - 点击"✗ 拒绝"按钮
   - 确认操作
   - 预期结果：提示"已拒绝家长请求"，请求从待处理列表移除

### 测试场景三：家长端状态更新

1. **刷新家长端页面**
   - 在学生接受请求后，刷新家长端"孩子管理"页面

2. **查看已添加的孩子**
   - 在"我的孩子"列表中应该能看到被接受的孩子
   - 显示孩子姓名、年级、学校等信息
   - 状态显示为"已添加"

3. **待处理请求变化**
   - 已接受的请求应该从"待处理的请求"列表中移除

## API 接口说明

### 家长端接口

1. **发送添加孩子请求**
   - POST `/api/parents/children/add`
   - 参数：`{ parentId, studentName, message }`
   - 返回：`{ success, message, request }`

2. **获取孩子列表**
   - GET `/api/parents/children/:parentId`
   - 返回：`{ success, children }`

3. **获取待处理请求**
   - GET `/api/parents/children/pending/:parentId`
   - 返回：`{ success, requests }`

### 学生端接口

1. **获取家长请求**
   - GET `/api/parents/children/student/requests/:studentId`
   - 返回：`{ success, requests }`

2. **接受请求**
   - PUT `/api/parents/children/requests/:requestId/accept`
   - 返回：`{ success, message, request }`

3. **拒绝请求**
   - PUT `/api/parents/children/requests/:requestId/reject`
   - 返回：`{ success, message, request }`

## 数据库模型

新增了 `ParentChildRequest` 模型，包含以下字段：
- `parent`: 家长ID（关联Parent模型）
- `student`: 学生ID（关联Student模型）
- `status`: 请求状态（pending/accepted/rejected）
- `message`: 请求消息
- `createdAt`, `updatedAt`: 时间戳

## 注意事项

1. **学生姓名必须真实存在**
   - 系统会验证输入的姓名是否在User表中存在且角色为student
   - 确保测试前数据库中已有学生用户

2. **不会破坏现有功能**
   - 新功能使用独立的ParentChildRequest模型
   - 不会影响原有的TeacherStudentMatch匹配功能
   - 教师端和学生端的匹配功能保持不变

3. **权限控制**
   - 所有接口都需要登录认证
   - 学生只能处理发送给自己的请求
   - 家长只能查看自己发送的请求

4. **实时性**
   - 当前实现需要刷新页面才能看到状态变化
   - 如需实时更新，可考虑添加WebSocket或轮询机制

## 常见问题

### Q: 提示"未找到该姓名的学生"
A: 请检查：
1. 学生是否已在数据库中注册
2. 输入的姓名是否与数据库中完全一致
3. 该用户的角色是否为'student'

### Q: 学生接受请求后家长端看不到
A: 请刷新家长端页面，当前需要手动刷新才能看到最新状态

### Q: 可以重复发送请求吗
A: 不可以，系统会检查是否已有pending状态的请求

### Q: 会影响老师-学生匹配功能吗
A: 不会，这是完全独立的功能，使用不同的数据表和接口
